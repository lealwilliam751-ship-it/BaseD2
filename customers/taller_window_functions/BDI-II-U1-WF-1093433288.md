## EJERCICIO 5. LAG
## a)
```sql
WITH monthly_sales AS (
    SELECT 
        DATE_TRUNC('month', T1.order_date) AS month,
        SUM(T1.total) AS current_month_sales
    FROM pay.orders T1
    GROUP BY DATE_TRUNC('month', T1.order_date)
)
SELECT 
    month,
    COALESCE(current_month_sales, 0) AS current_month_sales,
    COALESCE(LAG(current_month_sales) OVER (ORDER BY month), 0) AS prev_month_sales,
    COALESCE(current_month_sales, 0) - COALESCE(LAG(current_month_sales) OVER (ORDER BY month), 0) AS difference,
    ROUND(
        CASE 
            WHEN LAG(current_month_sales) OVER (ORDER BY month) = 0 OR LAG(current_month_sales) OVER (ORDER BY month) IS NULL THEN 0
            ELSE ((current_month_sales - LAG(current_month_sales) OVER (ORDER BY month)) * 100.0 / LAG(current_month_sales) OVER (ORDER BY month))
        END,
        2
    ) AS pct_change
FROM monthly_sales
ORDER BY month;
```

# EJERCICIO 6. LEAD — Tiempo hasta el próximo envío

## a)

```sql
WITH shipments_with_next AS (
    SELECT
        T1.ship_company_id,
        T1.order_id,
        T1.created_at,
        LEAD(T1.created_at) OVER (
            PARTITION BY T1.ship_company_id
            ORDER BY T1.created_at ASC, T1.id ASC
        ) AS next_shipment_date
    FROM ship.shipment_orders T1
)
SELECT
    ship_company_id,
    order_id,
    created_at,
    next_shipment_date,
    ROUND(
        EXTRACT(EPOCH FROM (next_shipment_date - created_at)) / 86400.0,
        2
    ) AS days_until_next
FROM shipments_with_next
ORDER BY ship_company_id, created_at;
```

> **Nota sobre los datos:** Los envíos fueron cargados en 3 batches masivos (20/03, 22/03 y 23/03), donde todos los registros de cada batch comparten exactamente el mismo `created_at`. Por esta razón, la mayoría de `days_until_next` muestra **0.00** — es el resultado correcto dado que miles de órdenes fueron registradas al mismo segundo. La Window Function funciona correctamente: `LEAD` trae el timestamp del siguiente envío dentro de la partición de cada empresa, y la diferencia entre timestamps idénticos es necesariamente 0. Los registros que cruzan entre batches sí muestran diferencias reales (ej: el último del batch 20/03 con el primero del batch 22/03 muestra ~1.37 días de diferencia).

## b)

```sql
WITH shipments_with_next AS (
    SELECT
        T1.ship_company_id,
        T1.created_at AS shipment_date,
        LEAD(T1.created_at) OVER (
            PARTITION BY T1.ship_company_id
            ORDER BY T1.created_at ASC, T1.id ASC
        ) AS next_shipment_date
    FROM ship.shipment_orders T1
),
days_between AS (
    SELECT
        ship_company_id,
        ROUND(
            EXTRACT(EPOCH FROM (next_shipment_date - shipment_date)) / 86400.0,
            4
        ) AS days_until_next
    FROM shipments_with_next
    WHERE next_shipment_date IS NOT NULL
)
SELECT
    T2.name AS company_name,
    ROUND(AVG(T1.days_until_next)::numeric, 4) AS avg_days_between_shipment
FROM days_between T1
JOIN ship.ship_company T2 ON T1.ship_company_id = T2.id
GROUP BY T2.id, T2.name
ORDER BY avg_days_between_shipment ASC;
```

## d)

### LAG vs LEAD: ambas funciones permiten acceder a filas adyacentes, pero en direcciones opuestas:

**Con LAG (mirando hacia atrás):**
```sql
SELECT
    ship_company_id,
    created_at,
    LAG(created_at) OVER (
        PARTITION BY ship_company_id
        ORDER BY created_at ASC, id ASC
    ) AS prev_shipment_date   -- NULL en la PRIMERA fila de cada empresa
FROM ship.shipment_orders;
```

**Con LEAD (mirando hacia adelante):**
```sql
SELECT
    ship_company_id,
    created_at,
    LEAD(created_at) OVER (
        PARTITION BY ship_company_id
        ORDER BY created_at ASC, id ASC
    ) AS next_shipment_date   -- NULL en la ÚLTIMA fila de cada empresa
FROM ship.shipment_orders;
```

### La diferencia principal:
- **LAG** accede a la fila anterior. `NULL` aparece en la **primera** fila de cada partición.
- **LEAD** accede a la fila siguiente. `NULL` aparece en la **última** fila de cada partición.
- Ambas consultas devuelven el mismo número de filas que la tabla original — ninguna "colapsa" el resultado.

# EJERCICIO 8. Segmentación de clientes — Combinando funciones

## a)

```sql
-- CTE 1: solo agregados (GROUP BY)
WITH customer_stats AS (
    SELECT
        T1.id_number                       AS customer_id_number,
        COALESCE(SUM(T2.total), 0)         AS total_spent,
        COUNT(DISTINCT T2.id)              AS total_orders,
        MIN(T2.order_date)                 AS first_purchase,
        MAX(T2.order_date)                 AS last_purchase
    FROM cs.customers T1
    LEFT JOIN pay.orders T2
        ON T1.id_number = T2.customer_id_number
    GROUP BY T1.id_number
),
-- CTE 2: Window Function aplicada SOBRE la CTE anterior
customer_ranked AS (
    SELECT
        customer_id_number,
        total_spent,
        total_orders,
        first_purchase,
        last_purchase,
        ROW_NUMBER() OVER (
            ORDER BY total_spent DESC
        ) AS global_rank
    FROM customer_stats
)
SELECT
    T2.name AS customer_name,
    T1.customer_id_number,
    T1.total_spent,
    T1.total_orders,
    T1.first_purchase,
    T1.last_purchase,
    T1.global_rank,
    CASE
        WHEN T1.total_spent > 10000000 THEN 'VIP'
        WHEN T1.total_spent > 3000000  THEN 'Regular'
        WHEN T1.total_orders >= 1      THEN 'New'
        ELSE 'Inactive'
    END AS segment
FROM customer_ranked T1
JOIN cs.customers T2
    ON T1.customer_id_number = T2.id_number
ORDER BY T1.global_rank ASC;
```

> Se usan **dos CTEs separadas** siguiendo el patrón exigido por el taller: `customer_stats` calcula únicamente los agregados con `GROUP BY`, y `customer_ranked` aplica `ROW_NUMBER()` sobre el resultado ya agrupado. # EJERCICIO 9. Reporte integrador — La pregunta del jefe

### Preguntas a responder:
1. ¿Cuál fue la primera orden de cada cliente?
2. ¿Cuánto lleva gastado en total?
3. ¿En el mes más reciente **de ese cliente**, ¿se vendió más o menos que el mes anterior?

## a)

```sql
WITH first_order AS (
    -- ROW_NUMBER responde: ¿cuál fue la primera orden de cada cliente?
    SELECT
        customer_id_number,
        order_date AS first_order_date,
        total      AS first_order_total,
        ROW_NUMBER() OVER (
            PARTITION BY customer_id_number
            ORDER BY order_date ASC, id ASC
        ) AS rn
    FROM pay.orders
    WHERE customer_id_number IN ('10012568', '10024615936', '1002749014')
),
customer_totals AS (
    -- SUM GROUP BY responde: ¿cuánto lleva gastado en total?
    SELECT
        customer_id_number,
        SUM(total)         AS total_spent,
        COUNT(DISTINCT id) AS total_orders
    FROM pay.orders
    WHERE customer_id_number IN ('10012568', '10024615936', '1002749014')
    GROUP BY customer_id_number
),
monthly_by_customer AS (
    SELECT
        customer_id_number,
        DATE_TRUNC('month', order_date) AS month,
        SUM(total)                       AS monthly_revenue
    FROM pay.orders
    WHERE customer_id_number IN ('10012568', '10024615936', '1002749014')
    GROUP BY customer_id_number, DATE_TRUNC('month', order_date)
),
monthly_with_lag AS (
    -- LAG responde: ¿el mes más reciente de ESTE cliente vendió más o menos que el anterior?
    SELECT
        customer_id_number,
        month,
        monthly_revenue                              AS current_month_sales,
        LAG(monthly_revenue) OVER (
            PARTITION BY customer_id_number          -- LAG particionado por cliente
            ORDER BY month ASC
        )                                            AS prev_month_sales,
        ROW_NUMBER() OVER (
            PARTITION BY customer_id_number
            ORDER BY month DESC
        )                                            AS rn
    FROM monthly_by_customer
)
SELECT
    T4.name                                                    AS customer_name,
    T1.first_order_date,
    T1.first_order_total,
    T2.total_spent,
    T2.total_orders,
    T3.current_month_sales,
    COALESCE(T3.prev_month_sales, 0)                           AS prev_month_sales,
    T3.current_month_sales - COALESCE(T3.prev_month_sales, 0)  AS difference
FROM first_order T1
JOIN customer_totals  T2 ON T1.customer_id_number = T2.customer_id_number
JOIN monthly_with_lag T3 ON T1.customer_id_number = T3.customer_id_number
                         AND T3.rn = 1
JOIN cs.customers     T4 ON T1.customer_id_number = T4.id_number
WHERE T1.rn = 1;
```

> **Corrección clave:** el `LAG()` en `monthly_with_lag` ahora usa `PARTITION BY customer_id_number`, lo que garantiza que la comparación mensual sea individual por cliente. Cada cliente compara su último mes contra **su** mes anterior, no contra el global del negocio. Esto produce valores de `current_month_sales`, `prev_month_sales` y `difference` distintos para cada cliente.

![Ejercicio 9](screenshots/E9a.png)

## b)

### Documentación de las CTEs:


**1. CTE `first_order`:**
- Usa **`ROW_NUMBER()`** para identificar la primera orden de cada cliente.
- `PARTITION BY customer_id_number` agrupa por cliente. `ORDER BY order_date ASC` ordena cronológicamente para obtener la más antigua.

**2. CTE `customer_totals`:**
- Usa funciones de agregación simples (`SUM`, `COUNT`) con `GROUP BY`.
- Calcula `total_spent` y `total_orders` por cliente.

**3. CTE `monthly_by_customer` + `monthly_with_lag`:**
- Agrupa ventas por cliente y mes, luego aplica **`LAG()`** particionado por cliente.
- Esto garantiza que cada cliente compare su mes más reciente contra **su propio** mes anterior, no contra el global del negocio.

## c)

### Versión sin Window Functions:

```sql
WITH first_order AS (
    -- Primera orden: subconsulta correlacionada en lugar de ROW_NUMBER
    SELECT
        o1.customer_id_number,
        o1.order_date AS first_order_date,
        o1.total      AS first_order_total
    FROM pay.orders o1
    WHERE o1.order_date = (
        SELECT MIN(o2.order_date)
        FROM pay.orders o2
        WHERE o2.customer_id_number = o1.customer_id_number
    )
    AND o1.customer_id_number IN ('10012568', '10024615936', '1002749014')
),
customer_totals AS (
    SELECT
        customer_id_number,
        SUM(total)         AS total_spent,
        COUNT(DISTINCT id) AS total_orders
    FROM pay.orders
    WHERE customer_id_number IN ('10012568', '10024615936', '1002749014')
    GROUP BY customer_id_number
),
monthly_sales AS (
    SELECT
        customer_id_number,
        DATE_TRUNC('month', order_date) AS month,
        SUM(total)                       AS monthly_sales
    FROM pay.orders
    WHERE customer_id_number IN ('10012568', '10024615936', '1002749014')
    GROUP BY customer_id_number, DATE_TRUNC('month', order_date)
),
last_month_per_customer AS (
    -- Mes más reciente de cada cliente: MAX sin Window Function
    SELECT customer_id_number, MAX(month) AS last_month
    FROM monthly_sales
    GROUP BY customer_id_number
),
monthly_comparison AS (
    -- Comparación mes actual vs anterior: self-join en lugar de LAG
    SELECT
        m1.customer_id_number,
        m1.monthly_sales AS current_month_sales,
        m2.monthly_sales AS prev_month_sales,
        m1.monthly_sales - COALESCE(m2.monthly_sales, 0) AS difference
    FROM monthly_sales m1
    JOIN last_month_per_customer lm
        ON m1.customer_id_number = lm.customer_id_number
       AND m1.month = lm.last_month
    LEFT JOIN monthly_sales m2
        ON m1.customer_id_number = m2.customer_id_number
       AND m2.month = m1.month - INTERVAL '1 month'
)
SELECT
    T4.name AS customer_name,
    T1.first_order_date,
    T1.first_order_total,
    T2.total_spent,
    T2.total_orders,
    T3.current_month_sales,
    COALESCE(T3.prev_month_sales, 0) AS prev_month_sales,
    T3.difference
FROM first_order T1
JOIN customer_totals      T2 ON T1.customer_id_number = T2.customer_id_number
JOIN monthly_comparison   T3 ON T1.customer_id_number = T3.customer_id_number
JOIN cs.customers         T4 ON T1.customer_id_number = T4.id_number;
```

# EJERCICIO 10. Diseño propio — Caso de uso libre

## a)

### Caso de uso propuesto:
**Análisis de frecuencia de compra por cliente:** determinar para cada cliente, ordenado por su fecha de compra, el tiempo transcurrido desde su compra anterior. Esto permite identificar clientes frecuentes vs clientes esporádicos y calcular el tiempo promedio entre compras por cliente para campañas de retención.

## b)

### Función a usar: **`LAG()`**

**Justificación:**
`LAG()` es la función adecuada porque permite acceder a la fecha de la orden anterior de cada cliente sin necesidad de hacer un self-join.

- **`PARTITION BY`:** `customer_id_number` — para agrupar las órdenes de cada cliente de forma independiente.
- **`ORDER BY`:** `order_date ASC` — para ordenar cronológicamente y que `LAG` obtenga la compra anterior en el tiempo.

## c)

```sql
WITH purchase_frequency AS (
    SELECT
        T1.customer_id_number,
        T2.name AS customer_name,
        T1.order_date,
        LAG(T1.order_date) OVER (
            PARTITION BY T1.customer_id_number
            ORDER BY T1.order_date ASC
        ) AS prev_order_date
    FROM pay.orders T1
    JOIN cs.customers T2 ON T1.customer_id_number = T2.id_number
)
SELECT
    customer_name,
    order_date,
    prev_order_date,
    ROUND(
        EXTRACT(EPOCH FROM (order_date - prev_order_date)) / 86400.0,
        2
    ) AS days_since_last_purchase
FROM purchase_frequency
WHERE customer_id_number IN ('10012568', '10024615936', '1002749014')
  AND prev_order_date IS NOT NULL
ORDER BY customer_name, order_date;
```

> **Corrección respecto a versión anterior:** se usa `EXTRACT(EPOCH FROM ...) / 86400.0` en lugar de `DATE_PART('day', ...)`. La resta de dos columnas `timestamp` en PostgreSQL devuelve un `interval`, y `DATE_PART('day', interval)` solo extrae la parte entera de días, truncando las horas. `EXTRACT(EPOCH FROM ...)` convierte el intervalo completo a segundos y dividir entre 86400 da los días con decimales precisos.

## e)

### Reflexión final:

Sin Window Functions, para resolver el mismo caso habría que:

1. Asignar un número de fila a cada orden por cliente mediante una subconsulta correlacionada que cuente cuántas órdenes anteriores tiene cada fila (simulando `ROW_NUMBER`).
2. Hacer un self-join de `pay.orders` consigo misma, uniendo cada orden (fila `n`) con la orden de número anterior del mismo cliente (fila `n-1`).
3. Calcular la diferencia de fechas entre las dos filas del join.
4. Filtrar las filas donde el número de posición del join sea exactamente una unidad menor.

**Con Window Functions:** la solución es más directa, legible y eficiente. Una sola pasada por los datos con `LAG()` reemplaza múltiples joins y subconsultas correlacionadas. La simplificación es significativa: de aproximadamente 20–25 líneas de código complejo a 10 líneas claras y mantenibles. Además, el self-join lee `pay.orders` dos veces; `LAG()` lo hace una sola vez.
